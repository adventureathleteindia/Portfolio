1. Executive Summary

This document outlines the business requirements for a Remote Patient Monitoring (RPM) platform that enables healthcare providers to monitor patient vitals remotely, conduct clinical reviews, and generate compliant billing claims. The platform will support Bluetooth-enabled medical devices, real-time data synchronization, and automated CPT code billing for Medicare reimbursement.

Business Problem

Healthcare providers lack an integrated solution to monitor chronic disease patients remotely, leading to missed interventions, hospital readmissions, and lost revenue from unbilled RPM services.

Target Users (B2B Multi-Tenant Architecture)

  • Super Admin: Platform administrators managing all providers, device inventory, and request queues
  • Healthcare Providers/Hospitals: Organizations enrolling patients, managing care teams, and tracking billing eligibility
  • Primary Ordering Providers (POP): Physicians responsible for patient care under RPM programs
  • Clinical Staff: Nurses reviewing patient vitals, adding notes, and managing alerts
  • PDC Billing Coordinators: Staff tracking eligibility, device readings, and CPT code qualification
  • Patients: Individuals with chronic conditions using BLE devices for remote monitoring

3. Scope & Boundaries

In Scope

  • Super Admin Dashboard: Central command for managing providers, devices, and requests
  • Multi-Provider Architecture: Support for multiple healthcare organizations with isolated data
  • Device Inventory Management: Track devices with ICCID/IMSI/IMEI, battery level, signal strength
  • Provider Management: Tabs for Provider Details, Allotted Devices, Patient Management, POP Management
  • Bulk Patient Onboarding: CSV upload with template download for multiple patient enrollment
  • External Care Provider Directory: Manage PCP and specialists with NPI numbers
  • Patient Health Statistics: Quick stats panel with vitals graphs and alert notifications
  • PDC Billing Module: Eligibility tracking, device reading counts, CPT code automation
  • BLE Device Integration: Blood pressure monitors, pulse oximeters, weight scales, glucometers
  • Request Management: Device allocation, PDC Nursing, PDC Billing request workflows
  • Activity Logs: Comprehensive audit trail for compliance
  • Patient mobile application for data sync
  • HIPAA-compliant data handling

Out of Scope

  • Direct EHR integration (Phase 2)
  • Telehealth video consultations (separate module)
  • Insurance eligibility verification
  • Patient billing and payment collection

5. Functional Requirements

5.1 Super Admin Dashboard

ID Requirement Priority
FR-001 Dashboard shall display stats cards: Total Providers, Inactive Providers, Total POP, Total Patients High
FR-002 Dashboard shall display: Unallotted Devices (available/total), Device Allocation Requests, PDC Billing Requests, PDC Nursing Requests High
FR-003 Dashboard shall show weekly change indicators for all stats Medium
FR-004 System shall display New Requests table with columns: S.No, Provider Name, Request Type, Patient ID, Dates, Status, Action High
FR-005 System shall support request types: Device Allocation, Support, PDC Nursing (Enable/Disable), PDC Billing (Enable/Disable) High
FR-006 Dashboard shall display analytics charts for Patients Onboarded and Requests volume (weekly/monthly) Medium

5.2 Device Inventory Management

ID Requirement Priority
FR-007 System shall track device information: Device ID, Device Type, Battery Level (%), Signal Strength (%) High
FR-008 System shall store device network identifiers: ICCID, IMSI, IMEI numbers High
FR-009 Device modal shall display: Latest Reading with timestamp, Assigned Patient (name, age, gender, location, last active) High
FR-010 Device status shall be tracked as: Active, Inactive, Unassigned High
FR-011 Device allocation workflow: Provider request → Super Admin review → Approve/Reject → Auto-link to patient High

FR-013 Provider stats shall show: Total Patients, Patients Enrolled (active monitoring), weekly change indicators High

FR-015 System shall isolate patient data between providers (multi-tenant security) High

ID Requirement Priority

FR-020 Bulk enrollment shall support: CSV upload (drag-and-drop), validation feedback, POP ID mapping High
FR-021 Patient status shall be tracked as: Enrolled, Pending, Inactive High

ID Requirement Priority

ID Requirement Priority

ID Requirement Priority
FR-030 System shall support BLE pairing with FDA-cleared blood pressure monitors High
FR-031 System shall support BLE pairing with pulse oximeters, weight scales, and glucometers High
FR-032 System shall automatically sync readings when patient app is in foreground or background High
FR-033 System shall retry failed syncs and queue readings for later transmission Medium

5.8 PDC Billing Module

ID Requirement Priority
FR-034 Billing screen shall display: Eligibility Status (Eligible/Not Eligible), Month Selector, POP Associated High
FR-035 Billing screen shall show Calls Taken count for clinical interactions High
FR-036 Per-device billing data shall include: Device Label (BPM Device 1, 2), Device ID, Device Status, Readings Received count High
FR-037 System shall calculate eligible CPT codes (99453, 99454, 99457, 99458) based on readings and clinical time High
FR-038 System shall generate monthly billing reports with patient-level detail High
FR-039 System shall export billing data in CSV format for claims submission High
FR-040 System shall track billing history and prevent duplicate claims Medium

7. Compliance Requirements (HIPAA)

HIPAA Compliance is Mandatory

All system components handling Protected Health Information (PHI) must comply with HIPAA Security Rule and Privacy Rule requirements.

Requirement Implementation
Access Control Role-based access with unique user IDs, automatic session timeout after 15 minutes of inactivity
Audit Controls Comprehensive audit logs for all PHI access, modification, and deletion events
Integrity Controls Data validation on input, checksums for data integrity verification
Transmission Security TLS 1.3 encryption for all data in transit, certificate pinning in mobile app
Encryption AES-256 encryption for PHI at rest in database and backups
BAA Requirements Business Associate Agreements with all third-party service providers

8. Sample User Stories

US-001: Patient Enrollment
As a clinical coordinator, I want to enroll a new patient into the RPM program, so that I can begin monitoring their vitals remotely.
Acceptance Criteria:
- Can enter patient demographics (name, DOB, contact info)
- Can select diagnosis codes from ICD-10 dropdown
- Can capture digital consent signature
- System assigns unique patient ID upon save
- Patient appears in active patient list immediately
US-002: Device Pairing
As a patient, I want to pair my blood pressure monitor with the mobile app, so that my readings are automatically sent to my care team.
Acceptance Criteria:
- App detects nearby BLE devices
- Patient selects device from list to initiate pairing
- Successful pairing confirmed with visual feedback
- Test reading syncs within 10 seconds
- Device remains paired across app restarts
US-003: Critical Alert Review
As a nurse, I want to see patients with critical readings highlighted, so that I can prioritize urgent interventions.
Acceptance Criteria:
- Patients with readings outside threshold appear in red
- Alert badge shows count of unreviewed critical readings
- Clicking patient opens detail view with vitals history
- Can mark alert as reviewed with clinical note
- Review time logged for billing purposes
US-004: Monthly Billing Report
As a billing manager, I want to generate a monthly RPM billing report, so that I can submit accurate claims for reimbursement.
Acceptance Criteria:
- Report shows all patients with qualifying activity
- CPT codes auto-calculated based on transmission days and clinical time
- Report includes patient name, MRN, dates of service, CPT codes
- Can export to CSV for upload to billing system
- Report excludes patients already billed for the period

9. Assumptions & Dependencies

Assumptions

  • Patients have smartphones capable of running the mobile app (iOS 14+ or Android 10+)
  • Clinical staff have access to modern web browsers (Chrome, Safari, Edge)
  • Internet connectivity is available at patient homes for data sync
  • Organization will provide FDA-cleared BLE devices to patients
  • Clinical staff will complete training before go-live

Dependencies

  • Device vendor API documentation for BLE integration
  • IT infrastructure provisioning (cloud hosting, SSL certificates)
  • Legal review and approval of patient consent forms
  • Compliance team sign-off on HIPAA security controls

10. Risks & Mitigations

HIPAA compliance gaps High Low Engage compliance team throughout; conduct security audit before launch
Billing calculation errors Medium Medium Validate billing logic with CMS guidelines; manual review in initial months
